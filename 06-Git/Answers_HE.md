# תשובות ראיון - Git & GitHub

**ש1. מה זה Git ולמה משתמשים בו?**
Git הוא מערכת בקרת גרסאות מבוזרת שעוקבת אחרי שינויים בקוד מקור. היא מאפשרת למספר מפתחים לעבוד על אותו פרויקט במקביל, שומרת היסטוריה מלאה של שינויים, ותומכת בהסתעפות (branching) ומיזוג (merging) לפיתוח מקבילי.

**ש2. מה ההבדל בין Git ל-GitHub?**
Git הוא כלי בקרת גרסאות מקומי שרץ על המחשב שלך. GitHub הוא פלטפורמת אחסון מבוססת ענן למאגרי Git שמוסיפה יכולות שיתוף פעולה כמו בקשות משיכה (Pull Requests), מעקב אחרי באגים (Issues), ובקרת גישה.

**ש3. מה זה מאגר (repository) ב-Git?**
מאגר (repo) הוא תיקייה שמנוהלת על ידי Git. היא מכילה את כל קבצי הפרויקט בתוספת תיקייה מוסתרת `.git` שמאחסנת את כל ההיסטוריה, הענפים וההגדרות. מאגרים יכולים להיות מקומיים (על המחשב שלך) או מרוחקים (על GitHub, GitLab וכו').

**ש4. מה ההבדל בין `git init` ל-`git clone`?**
`git init` יוצר מאגר חדש וריק בתיקייה הנוכחית. `git clone <url>` מעתיק מאגר מרוחק קיים למחשב המקומי שלך, כולל כל ההיסטוריה והענפים שלו.

**ש5. מהם שלושת השלבים (האזורים) ב-Git?**

1. **תיקיית העבודה (Working Directory)** — המקום בו עורכים קבצים.
2. **אזור ההכנה (Staging Area / Index)** — המקום בו מציבים שינויים מוכנים לקומיט (`git add`).
3. **המאגר (.git)** — המקום בו תמונות מצב מקובעות נשמרות לצמיתות (`git commit`).

**ש6. מה ההבדל בין `git add` ל-`git commit`?**
`git add` מעביר שינויים מתיקיית העבודה לאזור ההכנה. `git commit` לוקח את כל מה שבאזור ההכנה ושומר אותו כתמונת מצב קבועה בהיסטוריית המאגר.

**ש7. מה ההבדל בין `git pull` ל-`git fetch`?**
`git fetch` מוריד נתונים חדשים מהמאגר המרוחק אבל לא ממזג אותם לענף העבודה שלך. `git pull` עושה את שניהם — מביא את הנתונים ואז ממזג אוטומטית את השינויים המרוחקים לענף הנוכחי שלך. בעצם, `git pull` = `git fetch` + `git merge`.

**ש8. מה ההבדל בין `git merge` ל-`git rebase`?**
שניהם משלבים שינויים מענף אחד לאחר. `git merge` יוצר קומיט מיזוג חדש שמשלב את שני הענפים, ושומר על ההיסטוריה המלאה. `git rebase` מפעיל מחדש את הקומיטים שלך מעל ענף היעד, ומייצר היסטוריה ליניארית. מיזוג בטוח יותר לענפים משותפים; rebase נקי יותר לענפים מקומיים/פיצ'ר.

**ש9. מה זה קונפליקט מיזוג ואיך פותרים אותו?**
קונפליקט מיזוג מתרחש כש-Git לא יכול לשלב שינויים אוטומטית כי שני ענפים שינו את אותן שורות בקובץ. כדי לפתור: פותחים את הקובץ המתנגש, מחפשים את סימני הקונפליקט (`<<<<<<<`, `=======`, `>>>>>>>`), בוחרים את הקוד הנכון, מסירים את הסימנים, ואז מריצים `git add` על הקובץ ו-`git commit`.

**ש10. מה זה ענף (branch) ב-Git ולמה משתמשים בענפים?**
ענף הוא מצביע קליל לקומיט. ענפים מאפשרים למפתחים לעבוד על פיצ'רים, תיקוני באגים או ניסויים בבידוד מבלי להשפיע על קוד הבסיס הראשי. ברגע שהעבודה מושלמת ונבדקת, ניתן למזג אותה חזרה.

**ש11. איך יוצרים, עוברים ומוחקים ענפים ב-Git?**

- יצירה: `git branch <name>` או `git checkout -b <name>` (יצירה ומעבר).
- מעבר: `git checkout <name>` או `git switch <name>`.
- מחיקה מקומית: `git branch -d <name>` (בטוח) או `git branch -D <name>` (כפייה).
- מחיקה מרוחקת: `git push origin --delete <name>`.

**ש12. מה זה `HEAD` ב-Git?**
`HEAD` הוא מצביע לקומיט הנוכחי שאתה עובד עליו. בדרך כלל הוא מצביע לקומיט האחרון בענף הנוכחי. כשמחליפים ענפים, `HEAD` זז להצביע על קצה הענף החדש.

**ש13. מה זה מצב HEAD מנותק (detached HEAD)?**
HEAD מנותק אומר ש-`HEAD` מצביע ישירות לקומיט ספציפי במקום לענף. זה קורה כשעושים checkout לגרסת hash של קומיט או לתגית. כל קומיט חדש שנוצר במצב הזה לא שייך לאף ענף ויכול ללכת לאיבוד אם עוברים ענף בלי ליצור ענף חדש כדי לשמור אותו.

**ש14. מה ההבדל בין `git reset`, `git revert` ו-`git checkout`?**

- `git reset` מזיז את מצביע הענף אחורה, עם אפשרות להשליך קומיטים (משכתב היסטוריה).
- `git revert` יוצר קומיט חדש שמבטל קומיט קודם (בטוח לענפים משותפים, לא משכתב היסטוריה).
- `git checkout` מחליף ענפים או משחזר קבצים מקומיט ספציפי ללא שינוי ההיסטוריה.

**ש15. מהם שלושת המצבים של `git reset`?**

- `--soft` — מזיז את HEAD לקומיט היעד; שומר שינויים מוכנים (staged) ובתיקיית העבודה.
- `--mixed` (ברירת מחדל) — מזיז את HEAD; מבטל הכנה (unstage) אבל שומר שינויים בתיקיית העבודה.
- `--hard` — מזיז את HEAD; מוחק את כל השינויים מאזור ההכנה ומתיקיית העבודה.

**ש16. מה זה `git stash` ומתי משתמשים בו?**
`git stash` שומר זמנית שינויים שלא קובעו (גם מוכנים וגם לא מוכנים) ומחזיר את תיקיית העבודה למצב נקי. משתמשים בו כשצריך להחליף ענפים או למשוך עדכונים אבל לא מוכנים לבצע קומיט לעבודה הנוכחית. שחזור עם `git stash pop` או `git stash apply`.

**ש17. מה זה קובץ `.gitignore` ולמה הוא חשוב?**
קובץ `.gitignore` מפרט תבניות של קבצים ותיקיות ש-Git לא צריך לעקוב אחריהם — למשל תוצרי בנייה, `node_modules/`, קובצי `.env`, הגדרות IDE. הוא שומר על ניקיון המאגר ומונע קומיטים בטעות של קבצים רגישים או מיותרים.

**ש18. מה ההבדל בין מיזוג fast-forward למיזוג three-way?**
**מיזוג fast-forward** מתרחש כשלענף היעד אין קומיטים חדשים מאז שענף הפיצ'ר הסתעף — Git פשוט מזיז את המצביע קדימה, בלי צורך בקומיט מיזוג. **מיזוג three-way** מתרחש כששני הענפים התפצלו — Git יוצר קומיט מיזוג שמשלב את שני קווי הפיתוח.

**ש19. מה זה `git cherry-pick` ומתי הוא שימושי?**
`git cherry-pick <commit-hash>` מחיל קומיט ספציפי מענף אחד על ענף אחר בלי למזג את הענף כולו. שימושי להחלת תיקון חם (hotfix) מענף פיתוח לפרודקשן, או להכנסה סלקטיבית של שינויים בודדים.

**ש20. מה זה `git bisect` ואיך הוא עוזר לאתר באגים?**
`git bisect` מבצע חיפוש בינארי בהיסטוריית הקומיטים כדי למצוא איזה קומיט הכניס באג. מסמנים קומיט תקין ידוע וקומיט פגום ידוע, ואז Git עושה checkout לנקודת האמצע כדי שתבדקו. הוא מחצה את מרחב החיפוש בכל שלב, ומאתר ביעילות את הקומיט הבעייתי ב-O(log n) צעדים.

**ש21. מה ההבדל בין `git diff`, `git diff --staged` ו-`git diff HEAD`?**

- `git diff` — מציג שינויים בתיקיית העבודה ש**עדיין לא הוכנו** (not staged).
- `git diff --staged` (או `--cached`) — מציג שינויים ש**הוכנו אך עדיין לא קובעו** (staged but not committed).
- `git diff HEAD` — מציג **את כל השינויים** (גם מוכנים וגם לא) בהשוואה לקומיט האחרון.

**ש22. מה הם תגיות (tags) ב-Git? מה ההבדל בין תגית קלה (lightweight) לתגית מוערת (annotated)?**
תגיות מסמנות קומיטים ספציפיים, בדרך כלל לגרסאות שחרור (למשל `v1.0.0`). **תגית קלה** היא רק מצביע לקומיט (`git tag v1.0`). **תגית מוערת** מאחסנת מטא-דאטה נוסף — שם המתייג, תאריך והודעה (`git tag -a v1.0 -m "Release 1.0"`). תגיות מוערות מומלצות לגרסאות שחרור.

**ש23. מה זה בקשת משיכה (Pull Request) ואיך היא משתלבת בעבודת צוות?**
בקשת משיכה היא תכונה של GitHub שמאפשרת למפתח להודיע לצוות שענף פיצ'ר מוכן למיזוג. חברי הצוות סוקרים את הקוד, משאירים הערות, מבקשים שינויים ומאשרים. לאחר האישור, הענף ממוזג לענף הראשי. PRs משמשים כסקירת קוד ושער איכות.

**ש24. מה זה פורק (fork) ב-GitHub ומה ההבדל בינו לבין ענף (branch)?**
**פורק** הוא עותק מלא של מאגר תחת חשבון ה-GitHub שלך — משמש לתרומה לפרויקטים שאין לך הרשאת כתיבה אליהם. **ענף** הוא קו פיתוח מקבילי בתוך אותו מאגר. עושים פורק כדי להציע שינויים לפרויקט של מישהו אחר; עושים ענף כדי לעבוד בתוך הפרויקט שלך.

**ש25. מה זה GitHub Actions ואיך משתמשים בו ל-CI/CD?**
GitHub Actions הוא פלטפורמת אוטומציה מובנית. מגדירים תהליכי עבודה (workflows) בקובצי YAML (`.github/workflows/`) שמופעלים באירועים כמו push, בקשת משיכה או תזמון. שימושים נפוצים: הרצת בדיקות אוטומטית, בדיקת קוד (linting), בנייה ודחיפת Docker images, ופריסה לפרודקשן.

**ש26. מה זה Git hook? ציינו כמה hooks נפוצים ואת השימושים שלהם.**
Git hooks הם סקריפטים שרצים אוטומטית לפני או אחרי אירועי Git. הם נמצאים ב-`.git/hooks/`. hooks נפוצים:

- `pre-commit` — הרצת בודקי קוד (linters) או בדיקות לפני אישור קומיט.
- `commit-msg` — אכיפת פורמט הודעת קומיט.
- `pre-push` — הרצת בדיקות לפני דחיפה למאגר מרוחק.
- `post-merge` — התקנה אוטומטית של תלויות אחרי מיזוג.

**ש27. איך עושים squash לקומיטים ולמה עושים את זה?**
Squash משלב מספר קומיטים לאחד. משתמשים ב-rebase אינטראקטיבי: `git rebase -i HEAD~n`, ואז מסמנים קומיטים כ-`squash` (או `s`). זה שומר על היסטוריה נקייה על ידי כיווץ קומיטים רועשים של עבודה בתהליך לקומיט משמעותי אחד לפני מיזוג.

**ש28. מה זה `git reflog` ואיך הוא יכול לעזור לשחזר קומיטים שאבדו?**
`git reflog` מתעד כל תנועה של `HEAD` — החלפת ענפים, איפוסים, rebases, קומיטים. גם אחרי hard reset או rebase שנפל, הקומיט עדיין קיים ב-reflog למשך כ-90 יום. אפשר לשחזר אותו עם `git checkout <hash>` או `git reset --hard <hash>`.

**ש29. מה ההבדל בין `git rm` לבין מחיקת קובץ רגילה?**
מחיקת קובץ עם `rm` מסירה אותו מתיקיית העבודה אבל Git עדיין עוקב אחריו — תצטרכו להריץ `git add` על המחיקה. `git rm <file>` מסיר את הקובץ ומכין את המחיקה בשלב אחד, כך שהקומיט הבא מתעד את ההסרה.

**ש30. מה זה Git submodule ומתי משתמשים בו?**
Git submodule הוא מאגר מוטמע בתוך מאגר אחר. הוא מצמיד קומיט ספציפי של המאגר החיצוני. משתמשים ב-submodules כשהפרויקט שלך תלוי בפרויקט Git אחר (למשל ספרייה משותפת) ורוצים לעקוב אחרי גרסה ספציפית שלו באופן עצמאי.
